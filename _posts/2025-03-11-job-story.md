# Job story
> A job story describes user needs for software development. It focuses on context, motivation, and the expected outcome, which helps focus development on finding solutions with high utility.

**Job story** is a technique for describing user needs that emphasizes the situation, motivation, and expected outcome. Job stories focus on specific situations and real user behavior. They help better understand their actual needs and enable the design of more effective features. They prevent situations where feature development is based more on assumptions than on real user behavior.

![Job story template in English: When (situation) I want to (motivation) So I can (expected outcome).](https://substack-post-media.s3.amazonaws.com/public/images/189ac0ae-8d3e-4628-9418-3b07df9ae121_1024x268.png "1024x268")

[Job Story according to Alan Klement's article](https://jtbd.info/replacing-the-user-story-with-the-job-story-af7cdee10c27)

Each job story consists of three parts: The first part describes the situation in which the user encounters a problem. The second part states what the user wants to do to resolve the situation. The third part explains what benefit the solution provides for them.

For example, in a situation where a user receives a data message notification on their mobile phone, a job story might read:

> *When I receive a data message, I want to see a preview of the content to determine if I need to respond immediately.*

This format allows developers to focus solely on user behavior without having to predefine their role or demographic characteristics.

In software development, there are various methods for describing user needs. One of them is the user story, which is typically formulated in the format: *As a \[user profile\] I want \[action\] so that \[benefit\].* This approach focuses on user profiles, which can be useful when creating products for a specific user group, but sometimes leads to overly general or inaccurate assumptions.

Another approach is the *use case*, which details the individual steps of user interaction with the system. This method is very detailed and is best suited for technical documentation, but doesn't always help understand user motivations and needs. For example, if a user wants to speed up image loading on a slow connection, a use case might include steps like: (1) user enables data-saving mode, (2) system reduces image quality, and (3) content loads faster.

There is also an approach based on defining the problem (problem statement), which describes the current state, the problem, and its impact on the user. A problem statement is useful for problem analysis but doesn't offer a direct solution guide.

Job Stories are particularly useful in agile development and UX design, where understanding user behavior and designing products that respond to real situations is crucial. This approach allows developers to focus on aspects of the user experience that make sense in specific situations.
