# YAGNI: You Aren’t Gonna Need It
> YAGNI says: don't implement anything you don't need right now. Most pre-built features are never used—they only complicate development, maintenance, and code readability.

The [YAGNI](https://builtin.com/software-engineering-perspectives/yagni) principle ([You Aren’t Gonna Need It](https://www.meltstudio.co/post/engineering-principles-yagni), you won't need it) comes from [extreme programming](https://cs.wikipedia.org/wiki/Extr%C3%A9mn%C3%AD_programov%C3%A1n%C3%AD) and serves as a practical brake against premature complexity. The basic rule is: don't do anything extra until you have a real reason for it. Don't add configurations, interfaces, or architectural layers just because they "might come in handy someday." Most of the time, they won't.

For example: you have one payment gateway, but someone proposes creating a generic interface for potential future ones. A year passes, you're still using just one gateway—but now you have code nobody understands, which is harder to test and only complicates maintenance.

YAGNI does not advocate for bad code or postponing technical debt. It's about not inventing features without validating actual need. Code should be clean, but not over-engineered. Expand only when there's a real reason: a validated requirement, a concrete problem, or a new customer.

Experience shows that most "preventive code" either will never be needed or will have to be rewritten anyway. YAGNI thus helps limit waste, reduce technical debt, and keep development focused on current value.

Sticking to YAGNI is harder than it seems—ego, fear, and the desire to be prepared are strong. But that's precisely why we need it.
