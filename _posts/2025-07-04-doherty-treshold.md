# Doherty Threshold
> The usability boundary where a system is perceived as sufficiently fast by the user: 400 milliseconds.

The Doherty Threshold ([Doherty Threshold](https://lawsofux.com/doherty-threshold/)) denotes the boundary of approximately 400 milliseconds within which a system should respond to be perceived as instantaneous by the user. If the response falls within this window, the user remains focused and feels that everything is proceeding smoothly. Once the response is slower, it disrupts attention and increases frustration.

This term originated back in the 1980s thanks to research by Walter J. Doherty and Ahrvind J. Thadani. They discovered that a fast system response not only improves perception but directly [supports better user performance](https://jlelliotton.blogspot.com/p/the-economic-value-of-rapid-response.html). Later usability experts adopted the Doherty Threshold as a benchmark for designing responsive interfaces.

For web and application developers, this threshold is a clear goal: the system should respond quickly enough that the user doesn't even notice they are waiting. Techniques like optimistic UI updates, progressive content loading, or skeleton screens—which visually indicate system activity even during processing—help achieve this.

Application performance today is monitored by tools like [Web Vitals](https://www.vzhurudolu.cz/prirucka/web-vitals), which measure metrics such as time to first render or time to interactivity. When these values stay below the threshold, user trust increases and the risk of users abandoning the application decreases.

Meeting the Doherty Threshold makes an interface feel natural and snappy. Four hundred milliseconds can decide whether a user stays or leaves.
