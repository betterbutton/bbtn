# Point-of-need
> Point-of-need – a design approach that provides help precisely when and where the user needs it.

Designing for the point of need ([point‑of‑need](https://www.nngroup.com/articles/feature-checklists-are-not-enough/#:~:text=or%20common%20ones.-,Use%20Task%20Analysis%20to%20Identify%20the%20Point%20of%20Need,-The%20subtle%20problems)) is a design principle that ensures the user receives support exactly when they truly need it—not before and not after. This reduces frustration, increases task success, and maintains workflow fluidity. It could be subtle tooltips, automatic suggestions, or contextual tips—the key is that they do not disrupt the flow of work.

This approach has roots in librarianship and didactics, where the shift was made from static training to support directly during work. It later evolved into tools like EPSS and today is embodied by digital adoption platforms. These enable real-time display of advice, guides, or alerts directly within applications, based on what the user is currently doing.

For web managers and designers, this means moving help directly into the interface. For example: showing a help tip only when the user touches a form field. Or offering a tip upon an error, instead of displaying lengthy instructions in advance.

Best practices include progressive disclosure of information, behavior-triggered starters, and immediate feedback. The key is that the help is proportionate to the situation—not too much, not too little—and always available when the user seeks it.

When designing with point‑of‑need in mind, you are not just creating help. You are creating an interface that collaborates with the user, responds in real time to their needs, and actively supports their journey. For designers, it's a way of thinking—not an add-on, but a foundation.
